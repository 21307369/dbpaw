## 第一阶段 AI 增强计划（场景切换 + 方言约束 + 结构化输出 + schema 元数据增强）

### Summary
- 目标：把当前“单一 SQL 生成”升级为“三场景可切换”，并显著提高 SQL 可执行性与可控性。
- 本阶段只做低风险高收益改造，不引入 MCP，不改现有数据库驱动协议。
- 你已确认的策略：
1. `sql_explain` 输出“解释文本 + 优化建议”。
2. “假设列表”采用 SQL 顶部前置注释格式。
3. PK/FK/索引增强先覆盖“已选表”。

### Key Changes
- 场景切换（前端）
1. 在 AI 输入区增加场景选择器：`sql_generate | sql_optimize | sql_explain`，默认 `sql_generate`。
2. 发送请求时把场景值透传到后端，不再硬编码 `scenario: "sql_generate"`。
3. 场景切换后保留现有 provider、选表、会话逻辑不变。

- Prompt 模板升级（后端）
1. 将 prompt 版本升级（例如 `v1.1.x`），按场景拆分模板：
2. `sql_generate`：只返回 SQL，缺信息时允许前置注释输出 `-- Assumptions:`。
3. `sql_optimize`：只返回优化后 SQL，语义保持不变，必要时给 `-- Assumptions:`。
4. `sql_explain`：输出结构化解释文本（执行意图、关键条件、潜在风险、优化建议），不返回 SQL-only 约束。
5. 新增数据库方言约束段：基于连接 driver 注入（MySQL/Postgres/SQLite/MSSQL）语法规则与禁用语法提示。

- 输出结构化约束
1. 对 SQL 场景增加输出后处理校验：若出现 markdown code fence，自动剥离。
2. 对 SQL 场景增加“仅 SQL + 可选前置 assumptions 注释”的格式校验。
3. 当用户问题信息不足时，统一输出：
`-- Assumptions: ...`
`SELECT ...`

- schema 摘要增强（仅已选表）
1. 现有“已选表会实时拉结构”链路上，额外拉取表元数据（PK/FK/索引）。
2. prompt schema 摘要新增字段块：
- `Primary Keys`
- `Foreign Keys`
- `Indexes (unique/non-unique)`
3. 仍保留长度上限与截断策略，避免 token 爆炸。
4. 未选表场景维持现状（列名/类型摘要），不做全库 metadata 拉取。

### Public Interfaces / Types
- `AIChatRequest`：不新增字段，继续复用 `scenario`。
- 前端 AI 组件：新增“场景选择”UI 状态与请求绑定。
- prompt 构建接口：新增 `driver`（或 dialect）入参，用于方言规则注入。
- schema 摘要结构：在内部 prompt 渲染层扩展 metadata 文本，不改变外部 API 响应格式。

### Test Plan
- 单元测试
1. 场景模板测试：三个场景命中不同指令与输出约束。
2. 方言注入测试：MySQL/Postgres/SQLite/MSSQL 各自命中对应规则。
3. SQL 输出清洗测试：包含 code fence 时能正确剥离。
4. schema 摘要测试：包含 PK/FK/索引时渲染稳定且可截断。

- 集成测试（AI 请求链路）
1. 切换场景后后端收到正确 `scenario`。
2. 选表时会拉 metadata 并进入 prompt；不选表不触发 metadata 拉取。
3. `sql_generate/sql_optimize` 返回满足“SQL-only + 可选 assumptions 注释”。
4. `sql_explain` 返回解释文本结构，不误走 SQL-only 规则。

- 验收场景（手工）
1. 同一问题在四种方言下生成 SQL 关键语法正确（LIMIT/TOP、日期函数、标识符引用差异）。
2. 复杂关联问题中，模型更稳定命中主外键关系，不再频繁错连表。
3. 信息不全问题中，回答含明确 assumptions，而不是直接猜字段名。

### Expected Effects
- 可执行性提升：SQL 场景输出中 markdown/code fence 噪音显著下降，复制即执行成功率提升。
- 正确性提升：方言错误（如函数/分页语法混用）明显下降。
- 关系推理提升：由于 PK/FK/索引上下文增强，Join 路径与过滤建议更可信。
- 可解释性提升：`sql_explain` 输出稳定为结构化解释，便于审阅与二次提问。

### Assumptions
1. 第一阶段不引入 MCP、RAG、工具调用编排。
2. 不新增数据库层通用 schema metadata API，先复用“已选表逐表获取 metadata”。
3. 若当前连接缺失 driver 信息，则降级为通用 SQL 规则模板。
