**日常开发（快反馈）**
1. 改前端纯逻辑后：`bun run test:unit`
2. 改前端接口/命令调用后：`bun run test:service`
3. 改 Rust 后端逻辑后：`bun run test:rust:unit`
4. 提交前最小门禁：`bun run test:service && bun run test:rust:unit && bun run build`

**改数据库驱动相关（你现在最常见）**
1. 先跑基础：`bun run test:rust:unit`
2. 再跑对应驱动集成：
- MySQL: `RUN_MYSQL_IT=1 bun run test:integration`
- PostgreSQL: `RUN_POSTGRES_IT=1 bun run test:integration`
- SQLite: `RUN_SQLITE_IT=1 bun run test:integration`
- MSSQL: `RUN_MSSQL_IT=1 bun run test:integration`
- ClickHouse: `RUN_CLICKHOUSE_IT=1 bun run test:integration`

**上线前（全量建议）**
1. `bun run test:all`
2. 全驱动集成全开：
`RUN_MYSQL_IT=1 RUN_POSTGRES_IT=1 RUN_SQLITE_IT=1 RUN_MSSQL_IT=1 RUN_CLICKHOUSE_IT=1 bun run test:integration`
3. 最后再跑一次构建：`bun run build`

**实用节奏（单人高效）**
1. 开发中反复：`test:unit` / `test:rust:unit`
2. PR/合并前：`test:all`
3. 发版前：全驱动 `test:integration` + `build`



有，当前这套自动化已经覆盖了核心逻辑，但下面这些仍需要你人工测：

1. Tauri 前后端联调链路  
- 真正从前端点击触发 `invoke` 到 Rust 命令返回（不是 mock）。  
- 重点看错误提示是否正确展示、loading/重试状态是否正确。

2. UI 交互与体验  
- 表格编辑、复制 SQL、元数据面板、分页/排序/过滤在真实页面上的行为。  
- 这些自动测不到视觉和交互细节（错位、卡顿、状态不同步）。

3. 跨平台行为  
- macOS/Windows/Linux 的窗口行为、菜单、快捷键、路径兼容。  
- 你现在本地测到的，不等于其他系统都一致。

4. SSH 隧道与真实网络异常  
- 断网、抖动、代理、防火墙、远程 DB 超时。  
- 自动测试很难稳定复现真实网络环境。

5. 自动更新与安装流程  
- 下载更新、校验、安装、重启、回滚。  
- 这类发布链路必须人工端到端验证。

6. 文件系统与权限场景  
- SQLite 文件权限不足、只读目录、锁文件、路径含中文/空格/超长路径。  
- 部分有覆盖，但系统权限弹窗和真实用户目录仍需手测。

7. 性能与大数据量体验  
- 大表滚动、导出、首次加载时间、内存占用。  
- 单测/集成测主要测正确性，不测用户体感性能。

8. 老数据升级（迁移）  
- 从旧版本数据库升级到新版本的真实用户数据兼容。  
- 需要拿一份历史数据做升级演练。
